# v0.15.4 Report - Specifica RunningAction MoveTo e cause fallimento

## Stato

COMPLETATO / SPECIFICA OPERATIVA

---

## Obiettivo

Definire il contratto operativo della futura `RunningAction MoveTo` prima di implementare il movimento multi-cella produttivo.

Questa specifica nasce dopo tre fatti tecnici gia' emersi:

- `v0.15.1` ha corretto la cadenza decisionale quando una decisione non produce davvero un Job;
- `v0.15.3` ha corretto il caso target cibo sparito durante il movimento verso un job `EatKnownFood`;
- il movimento ordinario usa ancora `MoveIntent` e `MovementSystem` per path lunghi, macro-route, local search, porte e fallback legacy.

La fase `v0.15.4` non implementa ancora il nuovo movimento. Serve a rendere non ambigua la prossima patch.

## Contratto desiderato per RunningAction MoveTo

La futura `RunningAction MoveTo` deve diventare il percorso ordinario del movimento Job.

Deve ricevere:

- `npcId`;
- `jobId`;
- fase e azione correnti;
- cella di partenza;
- cella obiettivo;
- eventuale `targetObjectId`;
- route gia' lecita o risolvibile localmente;
- configurazione movimento;
- accesso al `World` solo per validazioni fisiche locali e mutazione finale controllata.

Deve produrre:

- progresso multi-tick;
- posizione aggiornata solo a completamento del singolo attraversamento;
- failure esplicito quando non puo' proseguire;
- trace EL leggibile;
- nessuna scelta di nuovo obiettivo.

Non deve produrre:

- nuova decisione cognitiva;
- nuovo target cibo;
- ricerca attiva globale;
- scansione onnisciente del mondo;
- fallback greedy nascosto;
- comando diretto fuori dal Job runtime.

|---|---|---|---|
| Target cell mancante | TargetInvalid | `MoveMissingTargetCell` | FailJob / Escalate |
| Target oggetto sparito | TargetInvalid | `MoveFoodObjectMissing` | FindEquivalentTarget / FailJob |
| Stock cibo assente | TargetInvalid | `MoveFoodStockMissing` | FindEquivalentTarget / FailJob |

| Cella successiva occupata | PathBlocked | `MoveToCellTraversalDestinationOccupied` | WaitAndRetry / Repath |

| Cella non attraversabile | PathBlocked | `MoveToCellTraversalTargetBlocked` | Repath / FailPhase |

| Timeout traversal | Timeout | `MoveToTraversalTimedOut` | RetrySameAction / FailPhase |

## Regola sulle porte

La porta e' una micro-interazione fisica locale.

La futura `MoveTo` puo':

- riconoscere una porta nella prossima cella o sul bordo locale del passo;
- aprirla se e' localmente apribile;
- produrre trace EL;
- continuare il movimento al tick successivo.

La futura `MoveTo` non deve:

- scegliere un'altra destinazione per aggirare la porta;
- decidere se rubare, forzare o violare regole sociali;
- trasformare `JobExecutionSystem` in decisore normativo.

Se la porta e' bloccata, non apribile o richiede scelta sociale, `MoveTo` deve fallire con causa esplicita.

---

## Regola sul fallback greedy

Il fallback greedy finale non deve restare comportamento ordinario del movimento Job.

Nel modello futuro:

```text
Job -> MoveTo -> route lecita -> progress multi-tick -> completion
```

oppure:

```text
Job -> MoveTo -> route assente/bloccata -> failure -> matrice recovery
```

Non deve esistere:

```text
Job -> MoveTo -> non conosco route -> greedy onnisciente -> arrivo comunque
```

---

## Relazione con `job_recovery_policies.json`

`MoveTo` deve restituire failure kind leggibili dalla matrice.

La matrice deve poi decidere:

- retry locale;
- attesa;
- eventuale replan;
- target equivalente;
- fallimento fase;
- fallimento job;
- escalation futura verso decisione.

`MoveTo` non deve leggere direttamente politiche cognitive e non deve scegliere recovery complesse. Il suo compito e' produrre un fallimento fedele.

---

## Relazione con memoria e belief

`MoveTo` puo' produrre un fallimento che dimostra che il target operativo non e' piu' valido.

Non deve pero' gestire tutto il lifecycle delle credenze.

Il caso gia' coperto da `v0.15.3` e':

```text
cibo conosciuto sparisce durante movimento
->
MoveToCell fallisce
->
Job fallisce o recovery sostituisce target
->
belief food locale viene scartata se la contraddizione e' diretta
```

Il lavoro piu' ampio resta rinviato a `v0.16`:

- obsolescenza generale delle belief;
- osservatori che vedono uno stock esaurito;
- memoria da eventi needs;
- verifica locale delle credenze.

---

## Prossimo step consigliato

`v0.15.5` deve aggiornare o specializzare la matrice recovery movimento in `job_recovery_policies.json`.

Obiettivo minimo:

- rendere esplicite le policy per failure di movimento;
- distinguere target invalido, risorsa mancante, path bloccato, reservation negata, informazione insufficiente e accesso negato;
- preparare l'implementazione multi-cella senza cambiare ancora autorita' decisionale.

---

## Rischi residui

- `MovementSystem` resta ancora authority viva per path lunghi.
- `MoveIntent` resta ancora ponte ordinario per target non adiacenti.
- La route multi-cella non e' ancora una struttura dedicata della running action.
- Le strategie `Repath`, `FindAlternateCell`, `FailPhase` ed `EscalateToDecision` sono ancora principalmente vocabolario/policy, non tutte hanno comportamento produttivo completo.
- La belief lifecycle completa resta fuori da questa fase.
