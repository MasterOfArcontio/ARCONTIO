GESTIONE NPC SIGHT

Nuovi processi (overview architetturale)

1) Oggetti: interagibili vs non-interagibili

Idea: un WorldObjectInstance è sempre “un oggetto nel mondo” (verità).

Poi distinguiamo l’oggetto per capabilities (data-driven da ObjectDef), non per classi.

  • Occluder (muro/porta): oggetto che contribuisce alla occlusion cache e (opzionale) ha health/hardness.
  • Interactable (letto, workbench, food stock): oggetto che entra nella “memoria oggetti” degli NPC e abilita comandi.

Quindi: una ObjectDef avrà flags/proprietà tipo:

  • BlocksVision, BlocksMovement, VisionCost (per occlusione/LOS)
  • IsInteractable (entra in memoria)
  • HasHealth, MaxHp, Hardness (distruzione)

2) Occluder come WorldObjectInstance + OcclusionMap cache derivata

Fonte di verità: World.Objects (come già fai).

Cache derivata: World.OcclusionMap (grid) per query veloci:

  • Quando crei/distruggi/sposti un oggetto che blocca, aggiorni la cache.
  • TryGetOccluder(x,y) diventa “consulta la cache” (O(1)) e, se vuoi, ritorna anche l’objectId occludente.

Questo risolve il problema: oggi si nasconde con _world.SetOccluder(...), ma poi l’NPC mangia lo stesso perché la “conoscenza/decisione” sta usando FoodStocks/oggetti senza considerare LOS sugli occluder coerentemente. Se tutto passa dalla stessa OcclusionMap, LOS e FOV diventano consistenti.

3) Rendering

Le viste non devono conoscere “Occluder vs altro”.

Leggono World.Objects e risolvono SpriteKey da ObjectDef.

  • Walls/doors sono come gli altri: hanno SpriteKey (tile key).
  • Se si vuole “tile png prelevata da atlas”: la view fa mapping SpriteKey → sprite.

4) Distruzione: Health + Hardness

Si aggiunge un componente (store) tipo:

  • World.ObjectHealth[objectId] = new HealthComponent{ Hp, MaxHp, Hardness }

Un DamageSystem:

  • riceve eventi/comandi di danno (es. ApplyDamageCommand)
  • applica: se damage <= hardness → niente; altrimenti hp -= (damage - hardness)
  • a hp <= 0: rimuove World.Objects[objectId] e aggiorna OcclusionMap liberando la cella.

5) Memoria oggetti “sparsa” per NPC con capacità fissa

Questo è il modo “cheap” per fare conoscenza di risorse/oggetti.

Struttura per NPC:

  • array fisso di slot KnownObjectSlot[] di dimensione N (da config)
  • ogni slot ha: ObjectId (o DefId + cella), CellX/Y, LastSeenTick, Confidence01, OwnerKind/OwnerId (se noto), InteractableMask
  • indicizzazione: lineare (N è piccolo) + merge per chiave (defId+cella oppure objectId se stabile)

Encoding: ObjectPerceptionSystem produce ObjectSpottedEvent → MemoryEncodingSystem (o un KnownObjectsEncodingSystem) aggiorna KnownObjectsStore dell’NPC.

6) Pulizia memoria oggetti (v1)

Politica:

  • mai eliminare oggetti posseduti dall’NPC (owner = npcId)
  • tenere sempre “top-K utili” in base a:
    • recency (LastSeenTick)
    • utility (letto/food/workbench > chair)
    • distance (opzionale)
    • confidence

Un KnownObjectsMaintenanceSystem gira ogni X tick e:

  • compatta slot vuoti
  • se over-capacity (o per mantenere qualità): rimuove i peggiori non-owned

3️⃣ Occlusion Map

3.1 Definizione

La OcclusionMap è una cache derivata da World.Objects.

Non è sorgente di verità.

Contiene solo informazioni necessarie a:

  • Pathfinding
  • Line of Sight
  • Vision attenuation

3.2 Invariante

World.Objects → aggiorna → OcclusionMap
Mai il contrario.

3.3 Aggiornamento

Ogni volta che:

  • Spawn object
  • Remove object
  • Door cambia stato

Si chiama:

RebuildOcclusionAtCell(x,y)

4️⃣ Doors (Runtime State)

4.1 DoorStateComponent

enum DoorState { Closed, Open }

4.2 Regole

  • Closed:
    • BlocksMovement = true
    • BlocksVision = true
  • Open:
    • BlocksMovement = false
    • BlocksVision = false

4.3 Interazione

  • Toggle solo se NPC ortogonalmente adiacente
  • Auto-open durante MoveTo
  • Apertura costa 1 tick
  • No auto-close
  • Non si chiude se cella occupata

5️⃣ Vision System

5.1 FOV Standard

  • 4 orientamenti: N/E/S/W
  • Cono di 90° (±45°)

5.2 Cone Check Formule

Con:

dx = targetX - npcXdy = targetY - npcY

Facing North

dy >0 &&abs(dx) <= dy

Facing South

dy <0 &&abs(dx) <= -dy

Facing East

dx >0 &&abs(dy) <= dx

Facing West

dx <0 &&abs(dy) <= -dx

5.3 Pipeline Unica di Visibilità

Ogni “vedo X” passa da:

  1. Range Check
  1. Cone Check
  1. LOS (Bresenham → OcclusionMap)

API canonica:

boolCanSee(int npcId,int targetX,int targetY);

6️⃣ NPC Orientation

6.1 Facing

enum Facing { North, East, South, West }

6.2 Regole

  • MoveTo aggiorna Facing
  • Turn90 costa 1 tick
  • Scan = 4 rotazioni consecutive
  • Nessuna visione a 360° gratuita

7️⃣ Perception System

7.1 Trigger

Percezione avviene:

  • Dopo movimento
  • Dopo rotazione
  • Se PerceptionDirty (evento locale)

7.2 Cooldown

  • Max 1 percezione per NPC per tick
  • Cooldown minimo: 1 tick

7.3 Eventi che “svegliano”

  • NpcMoved
  • PredatorMoved / Spawned
  • DoorStateChanged
  • ObjectDestroyed
  • ObjectSpawned (rilevanti)
Wake ≠ visibilità.

La visibilità passa sempre da CanSee().

8️⃣ Resource Stack

8.1 Struttura

struct StackComponent {int ItemDefId;int Quantity;int MaxQuantity;
}

8.2 Regole

  • Attraversabile
  • Consumo su stessa cella
  • Merge automatico stesso tipo
  • Non blocca LOS

9️⃣ Movement Rules

9.1 Walkable

Una cella è walkable se:

!OcclusionMap.BlocksMovement
AND!NpcAtCell

9.2 Door During Move

Se nextCell è door chiusa:

  1. Open door (1 tick)
  1. Non avanzare in questo tick
  1. Tick successivo entra
