# v0.17a — Audit punti caldi runtime per NPC

## Stato

DONE / AUDIT DOCUMENTALE

## Obiettivo

Questo audit individua i punti del runtime che possono aumentare sensibilmente il costo quando cresce il numero di NPC.

Non sono state applicate ottimizzazioni. Lo scopo e' preparare `v0.17b` e `v0.17c`, cioe' lo strumento di misura leggero che dovra' dire con precisione quale sistema consuma tempo, memoria temporanea o lavoro per NPC.

---

## Vincolo fondamentale della fase

L'osservatorio costi runtime deve essere quasi gratuito quando e' disattivato.

Questo significa:

- nessuna costruzione stringhe;
- nessuna lista temporanea;
- nessun dizionario per tick;
- nessun JSONL;
- nessun conteggio per NPC;
- nessuna misura temporale;
- solo un controllo booleano o un riferimento nullo prima di entrare nel codice di misura.

## 1. Percezione oggetti

### File coinvolto

`Assets/Scripts/Core/Systems/Perception/ObjectPerceptionSystem.cs`

### Funzionamento attuale

Ogni tick il sistema:

- crea uno snapshot degli NPC;
- crea uno snapshot degli oggetti;
- per ogni NPC controlla la visione;
- per ogni oggetto interagibile controlla distanza, cono visivo e linea di vista;
- pubblica `ObjectSpottedEvent`;
- invalida belief di cibo mancante se la cella creduta e' visibile.

### Costo probabile

Il costo ordinario e':

```text
NPC x oggetti
```

Con DebugFovTelemetry attivo si aggiunge:

```text
NPC x celle nel raggio visivo
```

Con invalidazione belief cibo si aggiunge:

```text
NPC x belief food x food stock
```

### Rischio

Alto.

Questo e' uno dei primi punti da misurare perche' cresce subito quando si aggiungono NPC o stock di cibo.

### Contatori consigliati

- NPC valutati;
- oggetti valutati;
- oggetti scartati per distanza;
- oggetti scartati per cono;
- oggetti scartati per linea di vista;
- eventi `ObjectSpottedEvent` prodotti;
- belief food controllate;
- belief food scartate;
- celle FOV registrate.

## 2. Percezione NPC

### File coinvolto

`Assets/Scripts/Core/Systems/Perception/NpcPerceptionSystem.cs`

### Funzionamento attuale

Ogni tick il sistema:

- crea uno snapshot degli NPC;
- per ogni NPC osservatore controlla tutti gli altri NPC;
- applica distanza, cono e linea di vista;
- pubblica `NpcSpottedEvent`.

### Costo probabile

```text
NPC x NPC
```

### Rischio

Alto.

Il costo cresce quadraticamente. Con 2 NPC e' quasi invisibile; con 16 NPC diventa molto piu' evidente.

### Contatori consigliati

- coppie NPC valutate;
- coppie scartate per distanza;
- coppie scartate per cono;
- coppie scartate per linea di vista;
- eventi `NpcSpottedEvent` prodotti.

---

## 3. Codifica memoria

### File coinvolto

`Assets/Scripts/Core/Systems/MemoryEncodingSystem.cs`

### Funzionamento attuale

Dopo i sistemi del tick, `SimulationHost` svuota gli eventi in un buffer e chiama `MemoryEncodingSystem`.

Il sistema:

- legge gli eventi del tick;
- per ogni evento prova le regole di memoria;
- per ogni evento compatibile controlla i possibili testimoni;
- applica distanza, cono e linea di vista;
- aggiunge o fonde trace in `MemoryStore`;
- aggiorna belief tramite `BeliefUpdater`;
- produce trace EL se abilitate.

### Costo probabile

```text
eventi x regole memoria x NPC
```

Questo costo puo' diventare molto alto perche' gli eventi di percezione sono gia' prodotti per NPC.

Esempio semplice:

```text
10 NPC vedono 5 oggetti
= 50 eventi ObjectSpottedEvent

50 eventi x regole x 10 NPC testimoni
= centinaia o migliaia di controlli nello stesso tick
```

### Rischio

Molto alto.

Questo e' il punto piu' sospetto per il costo moltiplicativo nascosto.

### Contatori consigliati

- eventi letti;
- regole provate;
- regole compatibili;
- testimoni valutati;
- testimoni validi;
- linee di vista calcolate;
- trace inserite;
- trace fuse;
- trace scartate;
- belief aggiornate;
- trace EL prodotte.

## 4. Comunicazione token

### File coinvolto

`Assets/Scripts/Core/Systems/TokenEmissionPipeline.cs`

### Funzionamento attuale

Ogni tick la pipeline:

- pota i cooldown di condivisione;
- crea snapshot degli NPC;
- controlla coppie NPC vicine;
- verifica orientamento;
- legge le top trace del parlante;
- prova le regole di emissione;
- pubblica token.

### Costo probabile

```text
NPC x NPC
```

piu':

```text
coppie valide x top memorie x regole token
```

### Rischio

Alto.

Anche se il numero di token prodotti e' limitato, la ricerca delle coppie e la lettura delle memorie avvengono ogni tick.

### Contatori consigliati

- coppie NPC valutate;
- coppie entro raggio;
- coppie con orientamento valido;
- store memoria letti;
- trace considerate;
- regole token provate;
- token prodotti;
- cooldown controllati;
- cooldown potati.

## 5. Consegna token

### File coinvolto

`Assets/Scripts/Core/Systems/TokenDeliveryPipeline.cs`

### Funzionamento attuale

La pipeline:

- svuota `TokenBusOut`;
- controlla distanza e linea di vista;
- per `AlarmShout` usa una ricerca BFS entro raggio;
- degrada intensita' e affidabilita';
- pubblica nel bus in entrata;
- registra log debug token e balloon.

### Costo probabile

Dipende dal numero di token.

Per canali normali:

```text
token x linea di vista
```

Per allarmi:

```text
token x BFS entro raggio
```

### Rischio

Medio/alto.

Non e' necessariamente il primo collo di bottiglia, ma puo' diventarlo quando cresceranno rumor, allarmi e sistemi sociali.

### Contatori consigliati

- token in uscita;
- token consegnati;
- token scartati per distanza;
- token scartati per linea di vista;
- token scartati per intensita';
- BFS avviate;
- celle BFS visitate.

## 6. Query belief e decisione

### File coinvolti

- `Assets/Scripts/Core/Decision/DecisionOrchestratorSystem.cs`
- `Assets/Scripts/Core/Decision/DecisionCandidateGenerator.cs`
- `Assets/Scripts/Core/Beliefs/BeliefQueryService.cs`

### Funzionamento attuale

La decisione:

- rispetta la cadenza configurata;
- costruisce contesto;
- genera candidati;
- per candidati che richiedono belief target invoca `BeliefQueryService`;
- calcola score;
- seleziona il candidato;
- prova a produrre un job.

La query belief:

- scorre tutte le belief dello store;
- filtra categoria, stato e confidence;
- valuta i candidati con evaluator;
- copia i contributi del miglior candidato;
- produce trace EL se abilitata.

### Costo probabile

```text
NPC eleggibili x intent candidati x belief x evaluator
```

### Rischio

Medio.

La cadenza decisionale riduce il costo, ma un numero alto di belief per NPC puo' renderla pesante.

### Contatori consigliati

- NPC valutati;
- NPC saltati per job attivo;
- candidati generati;
- candidati rimossi;
- query belief eseguite;
- belief lette;
- candidati belief usabili;
- evaluator eseguiti;
- job prodotti;
- job non prodotti.

## 8. Job, recovery e running action

### File coinvolti

- `Assets/Scripts/Core/Jobs/JobExecutionSystem.cs`
- `Assets/Scripts/Core/Jobs/MoveToRunningActionDriver.cs`

### Funzionamento attuale

- copia gli NPC con job attivo;
- esegue lo step corrente;
- valuta recovery locale;
- applica state machine;
- produce trace EL job/step/running action se abilitate.

- prepara route diretta, landmark o esplorativa;

### Costo probabile

### Rischio

Medio/alto.

Non sembra il costo principale per NPC fermi, ma puo' diventarlo quando molti NPC camminano o riprovano recovery.

### Contatori consigliati

- step failed;
- recovery valutate;
- recovery applicate;
- route preparate;
- route dirette;

## Proposta tecnica per v0.17b

Introdurre un osservatorio runtime con default spento.

Quando e' spento:

```text
world.RuntimeCostObserver == null
```

I sistemi caldi devono fare solo:

```text
var observer = world.RuntimeCostObserver;
if (observer != null)
{
    ...
}
```

Questo evita:

- oggetti misura;
- `Stopwatch` istanziati;
- stringhe canale;
- dizionari per tick;
- liste per NPC;
- JSONL;
- lambda/delegate;
- costi nascosti.

Quando e' acceso:

- usa `Stopwatch.GetTimestamp()` o equivalente statico;
- usa enum numerici per area, non stringhe;
- accumula in array preallocati;
- campiona ogni N tick;
- misura per sistema prima del per-NPC dettagliato;
- produce JSONL solo se esplicitamente abilitato e tramite scrittura batchata.

## Proposta di misure minime v0.17c

Misure per sistema:

- durata totale;
- chiamate per tick;
- contatori dominio;
- eventi prodotti;
- eventi consumati;
- entita' valutate;
- scarti per gate;
- trace prodotte;
- job/step/recovery eseguiti.

Misure per NPC solo in seconda fase:

- costo percezione oggetti;
- costo percezione NPC;
- belief lette;
- query eseguite;
- job step eseguiti;
- path/route preparate;
- token emessi/ricevuti.

---

## Rischi se si ottimizza prima di misurare

- Ottimizzare il pannello mentre il costo reale e' in memoria.
- Ottimizzare pathfinding mentre il costo reale e' nella percezione.
- Spegnere debug utile senza capire se e' davvero il collo di bottiglia.
- Cambiare comportamento simulativo per ridurre costo apparente.
- Misurare troppo e peggiorare il frame rate.

---

## Conclusione

`v0.17a` conferma che il problema va affrontato con un osservatorio mirato e congelabile.

La prima implementazione non deve provare a risolvere tutti i costi. Deve rendere visibili:

- costo per sistema;
- costo per NPC;
- moltiplicatori nascosti;
- differenza tra tick simulativo e pannelli visuali;
- differenza tra runtime normale e debug acceso.

Il prossimo step consigliato e':

```text
v0.17b — Configurazione osservatorio con costo nullo quando spento
```
