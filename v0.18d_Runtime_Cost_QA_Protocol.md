# v0.18d - Protocollo QA profiler runtime

## Stato

PARZIALE / PROTOCOLLO PRONTO / MISURAZIONE RUNTIME DA ESEGUIRE

---

## Obiettivo

Validare in runtime reale l'impatto della fase `v0.18` sui costi principali emersi dal file JSONL dell'osservatorio costi.

La fase `v0.18` ha introdotto tre ottimizzazioni locali:

- indice spaziale a griglia per `ObjectPerceptionSystem`;
- indice per categoria nel `BeliefStore`, usato da `BeliefQueryService`;
- decadimento belief discreto per categoria, governabile da configurazione.

Questo protocollo serve a confrontare il comportamento pre/post patch con scenari ripetibili a numero crescente di NPC.

---

## Configurazione richiesta

Nel file `game_params.json` deve essere attivo l'osservatorio costi runtime.

La sezione interessata e':

```json
"runtime_cost_observer": {
  "enabled": true,
  "sample_interval_ticks": 20,
  "write_jsonl": true,
  "track_per_npc": true
}
```

Se la sezione non e' presente, va aggiunta sotto la configurazione diagnostica gia' esistente, rispettando la struttura corrente del file.

---

## Configurazione opzionale v0.18a

Per testare il budget della percezione oggetti:

```json
"perception": {
  "maxCandidateCellsPerNpcPerTick": 0,
  "maxObjectsPerNpcPerTick": 0
}
```

Valore `0` significa nessun limite.

Per il primo confronto si consiglia di lasciare entrambi a `0`, cosi si misura solo il vantaggio dell'indice spaziale senza introdurre tagli artificiali.

---

## Configurazione opzionale v0.18c

Per testare il decadimento discreto dei belief, il file `belief_decay_config.json` puo' includere:

```json
"foodDecayIntervalTicks": 5,
"restDecayIntervalTicks": 10,
"dangerDecayIntervalTicks": 20,
"socialDecayIntervalTicks": 20,
"ownershipDecayIntervalTicks": 20,
"situationDecayIntervalTicks": 10,
"structureDecayIntervalTicks": 50
```

Se questi campi mancano, il default e' `1` e il decadimento resta applicato ogni tick come prima.

La scala del decadimento viene compensata: se una categoria decade ogni 10 tick, il sistema applica una scala temporale equivalente a 10 tick.

## Scenario di prova

Eseguire la stessa scena con questi numeri di NPC:

| Scenario | NPC | Durata minima | Note |
|---|---:|---:|---|
| Baseline minima | 1 | 1000 tick | Controllo che non ci siano errori o freeze |
| Piccolo gruppo | 2 | 1000 tick | Verifica crescita iniziale |
| Gruppo medio | 4 | 1000 tick | Primo confronto utile |
| Gruppo osservato nel bug | 8 | 1000 tick | Scenario vicino ai dati JSONL gia' raccolti |
| Stress leggero | 16 | 1000 tick | Verifica scalabilita' |

Per ogni scenario:

- usare lo stesso numero indicativo di stock/oggetti;
- evitare modifiche manuali durante la finestra di misura;
- lasciare attivi gli stessi pannelli;
- salvare il JSONL prodotto dall'osservatorio costi.

---

## Contatori principali da confrontare

| Contatore | Interpretazione | Esito atteso dopo v0.18 |
|---|---|---|
| `ObjectPerceptionObjectChecks` | oggetti realmente controllati dalla percezione oggetti | forte riduzione rispetto allo scan globale |
| `ObjectPerceptionCandidateCells` | celle candidate visitate dall'indice spaziale | cresce con raggio visivo, non con numero oggetti globale |
| `ObjectPerceptionDebugFovCells` | celle diagnostiche FOV | deve restare sotto controllo |
| `BeliefQueryEntriesRead` | belief letti dalle query | deve scendere verso i soli belief della categoria richiesta |
| `BeliefQueryCandidatesUsable` | belief realmente candidabili | non deve sparire se il comportamento decisionale e' corretto |
| `BeliefDecayEntriesUpdated` | belief aggiornati dal decay | puo' diminuire se gli intervalli per categoria sono > 1 |
| `DecisionNpcEvaluations` | valutazioni decisionali NPC | deve crescere circa con gli NPC, non esplodere |
| `JobExecutionSteps` | passi job eseguiti | deve restare coerente con attivita' reale |

---

## Criteri di accettazione

La patch e' considerabile efficace se:

- `ObjectPerceptionObjectChecks` non cresce piu' come `NPC x oggetti globali`;
- `ObjectPerceptionCandidateCells` cresce in modo controllato con raggio visivo e numero NPC;
- `BeliefQueryEntriesRead` e' sensibilmente inferiore al numero totale di belief dell'NPC quando esistono piu' categorie;
- `BeliefDecayEntriesUpdated` puo' essere ridotto tramite intervalli configurati senza cambiare esito qualitativo;
- non compaiono errori di compilazione o eccezioni runtime;
- gli NPC continuano a prendere decisioni e produrre job;
- non vengono introdotti freeze o stalli nuovi.

---

## Criteri di rischio

La patch va considerata da riaprire se:

- il valore di `ObjectPerceptionObjectChecks` resta proporzionale al numero totale di oggetti;
- `BeliefQueryEntriesRead` resta vicino al totale di tutti i belief, non alla categoria;
- il decadimento discreto fa sparire belief troppo rapidamente rispetto alla scala configurata;
- il numero di NPC aumenta ma l'osservatorio costi non produce file JSONL;
- l'EL o i pannelli generano piu' costo della simulazione osservata;
- un NPC resta senza decisioni/job in modo nuovo rispetto al ramo precedente.

---

## Output atteso della prova

Per chiudere definitivamente `v0.18d` servono:

- un file JSONL per ogni scenario;
- una tabella con i valori finali dei contatori principali;
- una nota se il comportamento visivo e decisionale e' rimasto corretto;
- eventuale screenshot profiler Unity se emerge un nuovo picco.

---

## Stato chiusura

Il protocollo e' pronto.

La misurazione reale resta manuale perche' richiede una sessione Unity con scena avviata, numero NPC controllato e cattura dei file JSONL prodotti in `AppData/LocalLow`.
