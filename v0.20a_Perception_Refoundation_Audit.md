# ARCONTIO - v0.20a Perception Refoundation Audit

## Stato

```text
AUDIT / SPECIFICA STRUTTURALE
NESSUNA PATCH RUNTIME IN QUESTO STEP
```

---

## Obiettivo

La `v0.20` non e' piu' una semplice fase di percezione cadenzata.

La fase diventa una rifondazione controllata del tratto:

```text
movimento
+
percezione
+
dirty percettivo
+
aggiornamento strutture World
+
indici persistenti oggetti/NPC
```

L'obiettivo e' ridurre drasticamente il costo CPU evitando che ogni NPC esegua ogni tick tutta la pipeline percettiva.

Il nuovo sistema deve arrivare a questo comportamento:

```text
solo chi deve percepire percepisce,
quando deve percepire,
con range e cadenza legati allo stato,
con limite massimo di NPC percepiti per tick,
senza percezione fuori ciclo,
senza conoscenza onnisciente,
senza generare memoria da entita' solo monitorate.
```

---

## Regola fondamentale sulle strutture dati

Ogni struttura introdotta nella `v0.20` deve rispettare questa regola:

```text
ridurre il numero totale di operazioni
+
rendere ogni operazione piu' economica
+
ridurre allocazioni runtime
```

Preferire nei percorsi caldi:

```text
byte
short
ushort
int
array
buffer riusabili
liste compatte
indici numerici
flag booleani
enum piccoli
```

Evitare nei percorsi caldi:

```text
long non necessari
stringhe
LINQ
ToList
ToArray
Dictionary annidati se evitabili
HashSet per piccoli insiemi se basta lista compatta
classi temporanee
DTO per tick
allocazioni per evento percettivo non necessarie
```

Le stringhe possono esistere per:

```text
pannelli
JSONL
debug
report
```

ma non devono essere necessarie per decidere nel tick caldo.

## Ciclo runtime desiderato

La percezione deve restare centralizzata nel ciclo `SimulationHost`.

Non e' ammesso chiamare percezione direttamente da:

```text
SetFacing
SetNpcPos
World.CreateObject
World.DestroyObject
JobExecutionSystem
Command
DevTools
```

Questi punti devono solo aggiornare stato autorevole e dirty.

Il ciclo concettuale resta:

```text
1. MOVIMENTO
2. NpcLandmarkMemorySystem
3. LandmarkPerceptionSystem
4. NeedsDecaySystem
5. PERCEZIONE
6. MemoryEncodingSystem
7. ObjectMemoryMaintenanceSystem
8. MemoryDecaySystem
9. BeliefDecaySystem
10. JobExecutionSystem
11. DecisionOrchestratorSystem
```

La differenza e' nel blocco percezione.

Prima:

```text
ObjectPerceptionSystem -> tutti gli NPC ogni tick
NpcPerceptionSystem    -> tutti gli NPC ogni tick
LandmarkPerception     -> periodo separato
```

Dopo:

```text
PerceptionScheduler seleziona gli NPC da processare
ObjectPerceptionSystem processa solo quegli NPC
NpcPerceptionSystem processa solo quegli NPC
LandmarkPerceptionSystem rispetta lo stesso gate
```

## Dirty percettivo

Un NPC e' dirty quando la percezione precedente non e' piu' affidabile.

Cause dirty:

```text
NPC ha ruotato
NPC si e' spostato davvero di cella
oggetto osservato e' cambiato
NPC osservato e' cambiato
oggetto/NPC nuovo e' apparso entro range teorico
oggetto/NPC e' stato distrutto o spostato
porta osservata si e' aperta o chiusa
stato percettivo dell'NPC e' cambiato
job/fase richiede osservazione attiva
```

La dirty deve avere almeno:

```text
npcId
reason numerica compatta
tick
urgenza
pending flag
```

Le reason devono essere preferibilmente enum piccoli, non stringhe.

---

## Dirty normale e dirty urgente

Servono due livelli.

### Dirty normale

```text
L'NPC percepisce quando arriva il suo tick percettivo.
```

### Dirty urgente

```text
L'NPC percepisce nel prossimo blocco percezione disponibile,
anche se non sarebbe il suo tick percettivo ordinario.
```

Decisioni gia' prese:

```text
SetFacing -> dirty urgente dell'NPC stesso
SetNpcPos con cambio cella reale -> dirty urgente dell'NPC stesso
```

La percezione resta comunque nel blocco centralizzato.

---

## Observed e watched

La nuova mappa percettiva deve distinguere:

```text
observed = visto davvero
watched  = monitorato per invalidazione
```

### Observed

`observed` puo' produrre:

```text
ObjectSpottedEvent
NpcSpottedEvent
memoria
belief
learning landmark
```

### Watched

`watched` non deve produrre:

```text
memoria
belief
ObjectSpottedEvent
NpcSpottedEvent
```

Serve solo a sapere che, se quell'entita' cambia, l'osservatore potenziale deve diventare dirty.

---

## Area di interesse

La percezione reale usa:

```text
range
cono
linea di vista
```

La mappa watched usa:

```text
cono reale + margine di 1/2 celle
```

Motivo:

```text
un oggetto appena fuori cono che entra nel cono non deve essere perso.
```

Esempio:

```text
NPC guarda nord.
Oggetto e' appena fuori cono ma dentro margine.
NPC non lo vede davvero.
Nessuna memoria viene prodotta.
L'oggetto viene registrato come watched.
Se l'oggetto cambia o si muove, l'NPC diventa dirty.
```

---

## Nuove entita'

Quando un oggetto o NPC viene creato non esiste ancora una lista osservatori.

Decisione:

```text
dirty conservativo dei possibili osservatori entro range massimo teorico.
```

Procedura:

```text
1. entita' nuova in cella X,Y;
2. calcola range massimo tra stati percettivi;
3. trova NPC entro area teorica;
4. marca dirty quegli NPC.
```

Non serve fare subito:

```text
cono preciso
linea di vista
memoria
belief
```

Questo lavoro resta nel blocco percezione.

## SetNpcPos

`SetNpcPos` cambia la posizione reale dell'NPC.

Nuovo comportamento richiesto:

```text
se la cella cambia davvero:
  aggiorna GridPos;
  aggiorna indice persistente NPC;
  marca dirty urgente l'NPC stesso;
  marca dirty gli osservatori che guardavano quell'NPC.
```

Nel movimento multi-tick:

```text
tick intermedi -> nessun cambio cella -> niente dirty posizione
tick completion cella -> SetNpcPos -> dirty
```

---

## SetFacing

`SetFacing` cambia la direzione dello sguardo.

Nuovo comportamento richiesto:

```text
se facing cambia davvero:
  aggiorna NpcFacing;
  marca dirty urgente l'NPC stesso.
```

Non deve fare percezione immediata.

## Indici persistenti

La `v0.20` deve introdurre indici persistenti.

Non lasciare il lavoro a meta'.

### ObjectSpatialIndex

Tiene:

```text
zona/cella -> elenco objectId grounded
```

Aggiornato da:

```text
CreateObject
DestroyObject
TryPickUpObject
TryDropObject
load/init rebuild
```

### NpcSpatialIndex

Tiene:

```text
cella/zona -> elenco npcId
```

Aggiornato da:

```text
creazione NPC
cancellazione NPC
SetNpcPos
load/init rebuild
```

### Rebuild completo

Durante:

```text
inizio simulazione
caricamento salvataggio
ricostruzione mondo
```

fare:

```text
rebuild ObjectSpatialIndex
rebuild NpcSpatialIndex
clear PerceptionDependencyMap
mark all NPC dirty iniziale
```

## PerceptionScheduler

Serve un coordinatore leggero che decide chi percepisce nel tick.

Responsabilita':

```text
legge dirty state
legge stato percettivo
legge cadenza
applica limite massimo NPC per tick
mantiene pending deterministico
produce contatori debug
```

Non deve:

```text
generare memoria
generare belief
decidere intent
decidere job
fare pathfinding
mutare world oltre stato percettivo/dirty
```

Gli stati vanno in `game_params.json`.

Esempio:

```json

  "defaultState": "Idle",
  "maxNpcPerceptionUpdatesPerTick": 4,
  "states": {
    "Idle": {
      "cadenceTicks": 8,
      "visionRangeCells": 10
    },
    "Moving": {
      "cadenceTicks": 4,
      "visionRangeCells": 14
    },
    "Observe": {
      "cadenceTicks": 1,
      "visionRangeCells": 17
    },
    "Alert": {
      "cadenceTicks": 1,
      "visionRangeCells": 17
    },
    "Combat": {
      "cadenceTicks": 1,
      "visionRangeCells": 17
    }
  }
}
```

Il valore globale `npcVisionRangeCells` resta fallback.

## Limite massimo NPC per tick

Aggiungere:

```text
maxNpcPerceptionUpdatesPerTick
```

Comportamento:

```text
se 20 NPC devono percepire
e limite = 4
processa 4 NPC
gli altri restano pending
```

La coda deve essere deterministica.

Scelta consigliata:

```text
round-robin stabile
```

Visualizzare in alto vicino alle coordinate mouse:



## Rischi principali

### 1. Indici persistenti incoerenti

Se un oggetto viene modificato bypassando World, l'indice puo' diventare falso.

Mitigazione:

```text
audit mutazioni dirette
metodi World ufficiali
rebuild completo init/load
test coerenza indice
```

### 2. Watched scambiato per observed

Rischio di conoscenza onnisciente.

Mitigazione:

```text
watched non produce mai eventi memoria/belief
```

### 3. Dirty permanente

Troppi cambiamenti possono rendere dirty quasi tutti.

Mitigazione:

```text
metriche dirty/pending/skipped
limite max NPC per tick
stati percettivi sfalsati
```

### 4. Percezione fuori ordine

Percezione fuori ciclo romperebbe ordine con MemoryEncoding.

Mitigazione:

```text
nessuna percezione fuori ciclo
solo dirty fuori ciclo
```

### 5. Save/Load e DevTools

Possono bypassare strutture.

Mitigazione:

```text
rebuild completo dopo load
dev tools passano da World ufficiale
```

## Formula sintetica

La nuova `v0.20` deve trasformare ARCONTIO da:

```text
tutti guardano tutto ogni tick
```

a:

```text
solo chi deve guardare guarda,
con stato percettivo,
```
