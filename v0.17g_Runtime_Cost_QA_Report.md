# v0.17g - Scenario QA costi runtime scalari

## Stato

```text
PARZIALE / RICHIEDE PROVA RUNTIME IN UNITY PROFILER
```

---

## Obiettivo

Validare l'osservatorio costi runtime introdotto in `v0.17` su scenari con numero crescente di NPC:

```text
1 NPC
2 NPC
4 NPC
8 NPC
16 NPC
```

Lo scopo non e' ottimizzare subito, ma capire quale sottosistema cresce peggio quando aumenta il numero di NPC.

---

## Configurazione richiesta

Per la prova manuale impostare temporaneamente:

```json
"runtime_cost_observer": {
  "enabled": true,
  "sampleEveryTicks": 1,
  "trackPerNpc": true,
  "writeJsonl": true,
  "maxTicksInMemory": 256,
  "jsonlFlushIntervalTicks": 25,
  "jsonlMaxQueueSize": 4096,
  "jsonlMaxBatchSize": 512,
  "jsonLogFileNamePattern": "arcontio_runtime_cost_{yyyyMMdd_HHmmss}.jsonl"
}
```

Al termine della prova riportare la configurazione al valore desiderato per il runtime ordinario.

## Canali da osservare

| Canale | Cosa misura | Rischio atteso |
|---|---|---|
| ObjectPerception | percezione oggetti e FOV debug | crescita NPC x oggetti |
| NpcPerception | percezione NPC verso NPC | crescita NPC x NPC |
| MemoryEncoding | eventi, regole, testimoni e trace memoria | crescita con eventi osservati |
| BeliefDecay | aggiornamento stato belief | crescita con numero belief |
| BeliefQuery | lettura belief e candidati decisionali | crescita con numero belief utili |
| Decision | valutazioni NPC e job avviati | crescita con NPC candidabili |
| JobExecution | step job, MoveTo, fallback locali | crescita con job attivi |
| TokenEmission | coppie sociali e token creati | crescita NPC x NPC |
| TokenDelivery | consegna token | crescita con token pendenti |
| TokenAssimilation | assimilazione token e trace aggiunte | crescita con token ricevuti |

---

## Contatori principali

| Contatore | Interpretazione |
|---|---|
| ObjectPerceptionObjectChecks | quanti oggetti vengono controllati dalla percezione |
| ObjectPerceptionDebugFovCells | quante celle vengono registrate dalla telemetria FOV |
| NpcPerceptionPairChecks | quante coppie NPC-NPC vengono valutate |
| MemoryEncodingRuleChecks | quante regole memoria vengono provate |
| MemoryEncodingWitnessChecks | quanti testimoni potenziali vengono controllati |
| BeliefQueryEntriesRead | quante belief vengono lette dalle query |
| JobExecutionSteps | quanti step job vengono consumati |
| MoveToExecutions | quante esecuzioni MoveTo vengono richieste |
| MoveToRoutePreparations | quante preparazioni route MoveTo vengono tentate |
| MoveToFailures | quanti fallimenti MoveTo vengono prodotti |
| JobRecoveryEvaluated | quante volte viene consultata la matrice recovery |
| JobRecoveryRetryScheduled | quante volte viene schedulato un retry |
| TokenEmissionPairChecks | quante coppie sociali vengono attraversate |
| TokenEmissionTokensCreated | quanti token vengono creati |

---

## Protocollo manuale consigliato

1. Avviare scena runtime neutra.
2. Abilitare `runtime_cost_observer`.
3. Eseguire 200-500 tick con 1 NPC.
4. Salvare screenshot profiler e file JSONL.
5. Ripetere con 2, 4, 8 e 16 NPC.
6. Confrontare crescita dei contatori rispetto al numero NPC.
